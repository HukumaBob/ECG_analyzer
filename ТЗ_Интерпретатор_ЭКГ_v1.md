# Техническое задание
# Система автоматической интерпретации ЭКГ на базе ECG-FM

**Версия:** 1.0
**Дата:** 2026-05-28
**Статус:** Черновик (первый этап)

---

## 1. Общие сведения

### 1.1 Наименование системы

**ЭКГ-Интерпретатор** — программная система автоматического анализа 12-канальных электрокардиограмм с формированием вспомогательного диагностического заключения для врача поликлиники.

### 1.2 Назначение

Система предназначена для поддержки принятия клинических решений: после снятия ЭКГ и до консультации врача система предоставляет предварительное заключение с вероятностными оценками по 17 диагностическим классам, визуализацией значимых участков сигнала и флагами критических находок.

**Система не является медицинским диагностическим прибором.** Все её выводы носят вспомогательный характер и не заменяют клиническое суждение врача.

### 1.3 Модель-основа

- **ECG-FM** (McKeen et al., *JAMIA Open*, 2025 / arXiv:2408.05178)
- Архитектура: wav2vec 2.0 (WCR pretraining), 90.9M параметров
- Обучена на: 2.5M ЭКГ (MIMIC-IV-ECG v1.0 + PhysioNet Challenge 2021 v1.0.3)
- Используемый чекпойнт: `mimic_iv_ecg_finetuned.pt` — fine-tuned для multi-label классификации по 17 классам
- Upstream-репозиторий: https://github.com/bowang-lab/ECG-FM

### 1.4 Этапность разработки

Настоящее ТЗ описывает **первый этап** — систему на базе готовой finetuned модели без дообучения. Второй этап (дообучение, Холтер, интеграция с МИС) описан в разделе 10.

---

## 2. Цели и ограничения

### 2.1 Цели системы

- Автоматически выявлять диагностически значимые паттерны ЭКГ из 17 стандартных классов
- Указывать, какие участки сигнала повлияли на каждый вывод (интерпретируемость)
- Корректировать вероятности с учётом клинического контекста пациента
- Формировать структурированный отчёт, читаемый врачом за 30–60 секунд
- Незамедлительно выделять критические находки (желудочковая тахикардия, инфаркт)

### 2.2 Ограничения первого этапа

- Только стандартные 12-канальные ЭКГ в покое (не Холтер, не нагрузочный тест)
- Классификация ограничена 17 обученными классами (расширение — второй этап)
- Без дообучения на локальных данных
- Без прямой интеграции с МИС (загрузка файла вручную)

## 3. Архитектура системы

### 3.1 Компонентная схема

```
Входной файл ЭКГ (12 каналов)
    │
    ▼
┌──────────────────────────────┐
│   Модуль предобработки       │
│  • конвертация формата       │
│  • ресемплинг → 500 Hz       │
│  • нормировка                │
│  • нарезка на 5-сек сегменты │
└──────────────┬───────────────┘
               │
    ┌──────────▼──────────┐
    │     ECG-FM          │
    │  (finetuned, 17 кл) │
    └──────────┬──────────┘
               │
    ┌──────────▼─────────────────┐
    │  Агрегация по сегментам    │
    │  (max / mean по классу)    │
    │  → P(class | ECG)          │
    │  → saliency maps           │
    └──────────┬─────────────────┘
               │
    ┌──────────▼─────────────────────────────────┐
    │  Модуль байесовской поправки               │
    │  + клинические метаданные (вход врача)     │
    │  → P'(class | ECG, metadata)              │
    └──────────┬─────────────────────────────────┘
               │
    ┌──────────▼─────────────────┐
    │  Модуль отчёта             │
    │  • диагностические метки   │
    │  • saliency на графике ЭКГ │
    │  • текстовое заключение    │
    │  • флаги тревоги           │
    └──────────┬─────────────────┘
               │
    ┌──────────▼──────────┐
    │  Веб-интерфейс      │
    │  врача              │
    └─────────────────────┘
```

### 3.2 Технологический стек (ориентировочный)

| Компонент | Технология |
|-----------|-----------|
| ML-бэкенд | Python 3.11, fairseq-signals, PyTorch |
| API-сервер | FastAPI или Flask |
| Веб-фронтенд | React или простой HTML/JS |
| Деплой | Локальный сервер поликлиники |

## 4. Требования к входным данным

### 4.1 Форматы файлов ЭКГ

Система должна принимать 12-канальные ЭКГ в следующих форматах:

| Формат | Расширение | Примечание |
|--------|-----------|-----------|
| WFDB | `.hea` + `.dat` | PhysioNet, ряд отечественных устройств |
| MATLAB | `.mat` | CODE-15, MIMIC-IV-ECG |
| CSV | `.csv` | Универсальный обменный формат |

**Открытые вопросы (требуют уточнения под конкретное оборудование поликлиники):**

- **DICOM-ECG** (`.dcm`) — формат устройств Welch Allyn, GE, Philips. Поддержку целесообразно добавить, если оборудование поликлиники его использует.
- **HL7 aECG** — используется в ряде МИС, поддержка зависит от интеграционных требований.

### 4.2 Технические параметры ЭКГ

- **Каналы:** обязательно 12 стандартных отведений (I, II, III, aVR, aVL, aVF, V1–V6)
- **Частота дискретизации на входе:** произвольная (система ресемплирует до 500 Hz)
- **Минимальная длительность записи:** 10 секунд (2 сегмента по 5 сек)
- **Рекомендуемая длительность:** 10–60 секунд

### 4.3 Клинические метаданные (вводятся врачом через форму)

| Поле | Тип | Обязательность |
|------|-----|----------------|
| Возраст | Число (лет) | Обязательно |
| Пол | М / Ж | Обязательно |
| ЧСС (из жалоб/анамнеза) | Число (уд/мин) | Опционально |
| Текущие препараты | Список (дигиталис, антиаритмики, β-блокаторы, ...) | Опционально |
| Активные МКБ-10 диагнозы | Структурированный список | Опционально |
| K+ (ммоль/л) | Число | Опционально |
| Mg²+ (ммоль/л) | Число | Опционально |

Если метаданные не введены, система выдаёт результат без поправки (на основании только ЭКГ).

## 5. Диагностические классы

Система классифицирует ЭКГ по 17 классам, используемым при обучении модели ECG-FM на датасете MIMIC-IV-ECG:

| № | Класс (EN) | Русское название | Клиническая значимость |
|---|-----------|-----------------|----------------------|
| 1 | Poor data quality | Низкое качество сигнала | Технический |
| 2 | Sinus rhythm | Синусовый ритм | Норма |
| 3 | Premature ventricular contraction | ЖЭС (желудочковая экстрасистолия) | Умеренная |
| 4 | Tachycardia | Тахикардия | Умеренная |
| 5 | Ventricular tachycardia | Желудочковая тахикардия (ЖТ) | **Критическая** |
| 6 | SVT with aberrancy | СВТ с аберрантным проведением | Высокая |
| 7 | Atrial fibrillation | Фибрилляция предсердий (ФП) | Высокая |
| 8 | Atrial flutter | Трепетание предсердий | Высокая |
| 9 | Bradycardia | Брадикардия | Умеренная |
| 10 | Accessory pathway conduction | Проведение по ДПП (синдром WPW и др.) | Высокая |
| 11 | Atrioventricular block | АВ-блокада (II–III ст.) | Высокая |
| 12 | 1st degree AV block | АВ-блокада I степени | Умеренная |
| 13 | Bifascicular block | Бифасцикулярная блокада | Умеренная |
| 14 | Right bundle branch block | Блокада ПНПГ | Умеренная |
| 15 | Left bundle branch block | Блокада ЛНПГ | Умеренная |
| 16 | Infarction | Инфаркт / инфарктоподобные изменения | **Критическая** |
| 17 | Electronic pacemaker | Электронный кардиостимулятор | Технический |

**Классы 5 (ЖТ) и 16 (Инфаркт)** при срабатывании выводятся в отдельном блоке с визуальным акцентом («флаг тревоги»).

## 6. Требования к байесовской поправке

### 6.1 Функциональное требование

Система должна корректировать апостериорные вероятности классов с учётом клинического контекста пациента.

**Входы модуля поправки:**
- Вектор вероятностей `P(class | ECG)` — 17 значений от ECG-FM
- Клинические метаданные пациента (возраст, пол, препараты, МКБ-диагнозы)

**Выход:**
- Скорректированный вектор вероятностей `P'(class | ECG, metadata)` — 17 значений

### 6.2 Логика поправки (примеры)

- Пациент получает дигиталис → повышается априорная вероятность нарушений ритма и АВ-блокады
- Возраст > 70 лет → повышается априорная вероятность АВ-блокад и блокад ножек пучка Гиса
- Известная ФП в анамнезе (МКБ I48) → повышается априорная вероятность ФП
- Ранее установленный кардиостимулятор → повышается априорная вероятность класса 17

Конкретный алгоритм реализации (наивный байес, таблица prior-смещений, экспертные правила) определяется на этапе проектирования.

## 7. Требования к выходным данным

### 7.1 Диагностические метки

- Список 17 классов с числовыми вероятностями `P'` (0–1)
- Классы, превышающие порог активации, помечаются как «выявлено»
- **Открытый вопрос:** конкретные пороги требуют клинической валидации (см. раздел 9)

### 7.2 Saliency на графике ЭКГ

- Для каждого активного класса — тепловая карта, наложенная на исходный ЭКГ-сигнал
- Выделяются временны́е интервалы и отведения с максимальным вкладом в прогноз
- Реализация: градиентные saliency maps через fairseq-signals (уже реализовано в upstream)

### 7.3 Текстовое заключение

Автоматически генерируемый текст на русском языке по шаблонам, формируемым на основе:
- Набора активных диагностических меток
- Значений метаданных (возраст, пол, препараты)

Пример структуры заключения:
```
Ритм: синусовый / нарушение ритма (...)
ЧСС: [значение] уд/мин
Проводимость: [нарушения / в норме]
Реполяризация: [изменения / в норме]
Заключение: [сводная фраза]
Примечание: анализ выполнен автоматически, требует верификации врачом.
```

### 7.4 Флаги тревоги

При выявлении критических классов (ЖТ, инфаркт) в интерфейсе отображается отдельный блок с явным визуальным выделением (цвет, значок). Цель — немедленно привлечь внимание врача при открытии отчёта.

## 8. Нефункциональные требования

### 8.1 Производительность

- Время от загрузки файла ЭКГ до отображения отчёта: не более 30 секунд на CPU, не более 5 секунд при наличии GPU
- Обработка одной записи не блокирует интерфейс

### 8.2 Деплой и инфраструктура

- Система разворачивается на **локальном сервере поликлиники** (без облачных зависимостей в production)
- ЭКГ и метаданные пациентов не передаются за пределы локальной сети
- GPU не является обязательным требованием (inference на CPU должен работать)

### 8.3 Интерфейс

- Веб-приложение, открывается в браузере (Chrome, Firefox) без установки клиентского ПО
- Поддерживает загрузку файла drag-and-drop и через кнопку «Выбрать файл»
- Форма ввода клинических метаданных — до заполнения отображается как опциональная
- Результат доступен на одной странице: форма ЭКГ с saliency + таблица меток + текст заключения

### 8.4 Надёжность

- При ошибке обработки (повреждённый файл, неподдерживаемый формат) система отображает понятное сообщение об ошибке без падения сервиса
- Класс 1 («Низкое качество сигнала») при высокой вероятности должен предупреждать врача о ненадёжности остальных выводов

## 9. Валидация системы

### 9.1 Датасеты для валидации

| Датасет | Доступность | Назначение |
|---------|------------|-----------|
| MIMIC-IV-ECG v1.0 | Требует DUA на physionet.org | Основная валидация (те же данные, что при обучении) |
| PTB-XL | Открытый (CC BY 4.0) | Внешняя валидация на европейской популяции |
| PhysioNet Challenge 2021 | Открытый | Мультицентровая валидация |

### 9.2 Метрики качества

- **AUROC** — для каждого из 17 классов отдельно
- **F1-score** (macro и per-class)
- **Чувствительность и специфичность** — особенно важны для критических классов (ЖТ, инфаркт)
- **Calibration** — насколько вероятности модели соответствуют реальной частоте классов

### 9.3 Открытые вопросы

- **Пороги активации** для каждого класса требуют клинической экспертизы и зависят от приоритета чувствительности vs. специфичности
- **Клиническая приемлемость** saliency-карт (понятны ли они врачу) — требует пользовательского тестирования
- **Валидация байесовской поправки** — нужно проверить, улучшает ли поправка метрики на валидационных выборках

## 10. Направления развития (этап 2)

| Направление | Описание |
|-------------|---------|
| **Холтер-мониторинг** | Поддержка EDF-формата, обработка длинных (24ч+) записей: скользящее окно, агрегация, подавление шумов |
| **Дообучение** | Fine-tuning или linear probing на локальных размеченных данных для адаптации к конкретной поликлинике |
| **Расширение классов** | Добавление диагностических классов сверх 17 базовых (через linear probing на новых данных) |
| **Интеграция с МИС** | Получение ЭКГ и метаданных напрямую через HL7 FHIR или HL7 v2 |
| **Улучшенное слияние метаданных** | Конкатенация клинических эмбеддингов с ЭКГ-эмбеддингами вместо байесовской поправки |
